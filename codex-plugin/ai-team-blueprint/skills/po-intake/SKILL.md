# po-intake (Getting Things Done, for product input)

Run this on ANY product input: an idea, a complaint, an observation, a feature
request, a strategy question. It is the mechanic that makes the product owner a
real gatekeeper rather than a title.

## The five stages

1. **Capture.** Write the input into the backlog immediately, verbatim enough to
   act on later. An idea that lives only in chat is lost. Never triage from
   recall.
2. **Clarify.** Ask: is this actionable? What job is the customer hiring us for
   (Jobs to be Done)? What outcome would prove the job is done better? If the
   honest smallest version is tiny, that is the slice. If it is not actionable,
   file it as reference or someday, not a task.
3. **Organize.** Place it: this sprint, next-up, someday/maybe (the archive), or
   reference. Attach a priority (see `score-initiatives`) and name its blocking
   dependencies and any hazard files.
4. **Reflect.** Trust the review cadence (daily review, the weekly outcome
   review, event-driven retro) to keep the backlog honest: stale items die,
   priorities re-rank against new evidence.
5. **Engage.** Pick the next action by priority and context. It enters the
   sprint ONLY as a groomed story with a measure of success.

## Output

The input, captured durably, with a clarify verdict (actionable slice /
reference / someday), a placement, and a priority. Or an honest clarifying
question back to the user when the frame is missing.
